# awd-c0iq 使用说明

本工具面向已获授权的 AWD/CTF 靶场排障、值守、自检和批量操作场景。不要在未授权目标上使用。

## 1. 先看整体使用顺序

建议按下面顺序使用：

1. 启动程序一次，让它自动生成 `config.yaml`、`output/`、`log/`、`pcap.sqlite`、`bleve/`、`pcap/`。
2. 配好 `config.yaml` 里的 Shell、SSH、数据库参数。
3. 准备目标列表，保存到界面里或直接写入 `output/target.txt`。
4. 先做“存活探测”和“Shell 测试”，确认链路可用。
5. 再按需要执行攻击链、取旗、防守、恢复、批量改密、PCAP 检索。
6. 需要回归时，最后跑“集成自检”。

## 2. 启动前要准备什么

### 本机环境

- Windows 主机，直接运行打包好的 `exe`。
- 如果要跑本地靶场自检，额外准备 WSL。
- Go 环境不是运行 GUI 的必需项，但“集成自检”会尝试重编译 `cmd/integration-smoke`，所以建议把 `go` 放进 `PATH`。

### 目标侧最少准备

- 一份目标 IP/域名列表。
- 一个可用的基础 WebShell，供“Shell 测试”和后续上传其他载荷使用。
- 如果要用防守、恢复、批量 SSH 改密，必须有目标机 SSH 账号。
- 如果要用数据库备份/恢复/改密，必须准备数据库账号，并且目标机上要能执行 `mysql` / `mysqldump`。
- 如果要用 PCAP 检索，需要准备 `.pcap` 或 `.pcapng` 文件。

## 3. 程序第一次启动后会生成什么

程序目录下会自动生成这些内容：

- `config.yaml`：主配置文件。
- `output/`：目标、测试结果、取旗结果、备份结果等输出文件。
- `log/`：运行日志。
- `pcap/`：上传的流量文件。
- `pcap.sqlite`：PCAP 元数据存储。
- `bleve/`：PCAP 全文索引。

`output/` 下默认会有这些文件：

- `target.txt`：目标列表。
- `success.txt` / `error.txt`：普通 Shell 测试成功/失败目标。
- `flag.txt`：取旗结果。
- `md5_horse_success.txt` / `md5_horse_error.txt`
- `undead_horse_success.txt` / `undead_horse_error.txt`
- `worm_paths.txt` / `worm_paths_all.txt`
- `ssh_password_success.txt`

## 4. config.yaml 怎么配

`config.yaml` 放在程序同目录，常用字段如下。

### 4.1 Shell

| 字段 | 作用 | 使用前要确认 |
| --- | --- | --- |
| `shell.port` | WebShell 端口 | 默认 `80`，非 80 要改 |
| `shell.pass` | WebShell 参数名 | 例如很多一句话马默认是 `pass` |
| `shell.path` | Web 根相对路径 | 例如 `/`、`/admin` |
| `shell.file` | WebShell 文件名 | 例如 `shell.php` |
| `shell.method` | 调用方式 | 支持 `POST` / `GET`，默认 `POST` |
| `shell.timeout` | HTTP 超时秒数 | 网络慢时适当调大 |
| `shell.proxy` | HTTP 代理 | 抓包转发时填写，例如 `http://127.0.0.1:8080` |

普通 Shell 实际请求地址形如：

```text
http://目标IP:shell.port/shell.path/shell.file
```

程序会把执行载荷放到 `shell.pass` 对应的参数里发出去。

### 4.2 SSH

| 字段 | 作用 | 使用前要确认 |
| --- | --- | --- |
| `ssh.host` | 默认 SSH 主机 | 仅在 `target.txt` 为空时兜底使用 |
| `ssh.port` | SSH 端口 | 默认 `22` |
| `ssh.username` | SSH 用户名 | 不能为空 |
| `ssh.password` | SSH 密码 | 不能为空 |
| `ssh.path` | 站点根目录 | 例如 `/var/www/html` |

### 4.3 Database

| 字段 | 作用 | 使用前要确认 |
| --- | --- | --- |
| `database.host` | 数据库主机 | 从目标机视角可连通 |
| `database.port` | 数据库端口 | 默认 `3306` |
| `database.username` | 数据库用户名 | 要有备份/恢复/改密权限 |
| `database.password` | 数据库密码 | 不能为空 |
| `database.name` | 数据库名 | 不能为空 |

### 4.4 Undead / Worm

| 字段 | 作用 | 使用前要确认 |
| --- | --- | --- |
| `undeadHorse.urlPass` | 备用参数 | 当前上传逻辑里主要还是依赖 `pass` |
| `undeadHorse.pass` | Undead 马密码 | 上传和执行时要一致 |
| `undeadHorse.filename` | Undead/MD5 落地文件名 | 默认 `favicon.php` |
| `wormShell.urlPass` | Worm 备用参数 | 当前执行逻辑里实际影响较小 |
| `wormShell.pass` | Worm 密码 | 要和执行阶段一致 |

## 5. 目标列表怎么准备

目标列表来源优先级是：

1. 界面里手工录入并保存。
2. `output/target.txt`。
3. 如果 `target.txt` 为空，部分 SSH 功能会回退到 `ssh.host`。

支持的输入格式：

- 单个 IP：`192.168.1.10`
- 一行一个目标
- 末位范围：`192.168.1.10-20`
- 完整范围：`192.168.1.10-192.168.1.20`

建议先保存目标，再跑“存活探测”。

## 6. 各功能使用前要准备什么

### 6.1 存活探测

准备：

- 目标列表。
- 目标至少有 ICMP、80、22、443 其中一个可通。

说明：

- 程序会先尝试 `ping`，失败后再探测 `80/22/443`。
- 探测出的存活主机会作为后续操作候选。

### 6.2 普通 Shell 测试

准备：

- 目标站点上已经存在一个可执行命令的基础 WebShell。
- `shell.port`、`shell.path`、`shell.file`、`shell.pass`、`shell.method` 填对。

说明：

- 测试命令是 `echo AWD_OK`。
- 成功目标会写到 `output/success.txt`。
- 失败目标会写到 `output/error.txt`。

### 6.3 上传 Undead / MD5 / Worm

这三项都不是“凭空打进去”，都依赖一个已经可用的基础 Shell。

#### Undead 马

准备：

- 普通 Shell 已测试通过。
- 当前 Web 根目录对 Shell 进程可写。
- `undeadHorse.pass`、`undeadHorse.filename` 配置正确。

说明：

- 会把 Undead 代码写到 `undeadHorse.filename`。
- 上传后会触发自恢复逻辑，再轮询确认是否可执行。
- 成功/失败会写到 `output/undead_horse_success.txt` 和 `output/undead_horse_error.txt`。

#### MD5 马

准备：

- 普通 Shell 已测试通过。
- Web 根目录可写。
- 你要明确执行时使用的 URL 密码、POST 字段名。

说明：

- 当前实现会把 MD5 马也落到 `undeadHorse.filename` 对应的文件名上。
- 成功/失败会写到 `output/md5_horse_success.txt` 和 `output/md5_horse_error.txt`。

#### Worm

准备：

- 普通 Shell 已测试通过。
- 目标站点 PHP 文件可写。
- 已提前做站点备份。

说明：

- Worm 会尝试修改目标目录下的 PHP 文件。
- 这是破坏性最高的功能，务必先备份。
- 结果路径会写到 `output/worm_paths.txt` 和 `output/worm_paths_all.txt`。

### 6.4 命令执行

准备：

- 你要先确认当前使用的是哪条链路：普通 Shell、Undead、MD5、Worm。
- 对应上传是否已经成功。

说明：

- 普通 Shell 直接使用 `shell.*` 配置。
- Undead 执行依赖 `undeadHorse.filename` 和 `undeadHorse.pass`。
- MD5 执行时要把 URL 密码、口令、POST 字段名与上传时保持一致。

### 6.5 HTTP 取旗

准备：

- 旗子能通过 HTTP 直接访问。
- 你知道 flag URL 或路径模板。

说明：

- 支持直接填完整 URL。
- 如果只填路径，例如 `/flag`，程序会自动按目标拼成 `http://目标IP:端口/flag`。
- 支持模板变量 `{ip}`、`{host}`、`%s`。
- 结果写到 `output/flag.txt`。

### 6.6 Shell 取旗

准备：

- 某条命令执行链路已可用。
- 你知道 flag 的读取方式，例如 `cat /flag`。

说明：

- 如果不填命令，默认会尝试读取你填的远程路径。
- 结果同样写到 `output/flag.txt`。

### 6.7 站点备份 / 恢复 / 巡检 / 加固

这些功能都依赖 SSH。

准备：

- `ssh.username`、`ssh.password`、`ssh.port`、`ssh.path` 正确。
- 目标机允许当前账号通过 SSH 登录。

#### 站点备份

说明：

- 远程执行 `tar -czf` 打包 `ssh.path`。
- 备份文件会下载到 `output/backup/web/`。

#### 站点恢复

准备：

- 先有一份已经备份好的站点压缩包。

说明：

- 会把本地备份重新上传到目标机，再解压到 `ssh.path`。
- 恢复会覆盖目标站点内容。

#### 查马

说明：

- 会在 `ssh.path` 下扫描 `php/phtml/jsp/asp/aspx` 等脚本文件。
- 规则主要看 `eval(`、`assert(`、`base64_decode`、`system(`、`shell_exec`、`passthru(` 等关键字。

#### 主机巡检

说明：

- 会采集 `hostname`、`whoami`、`uname -a`、`id`、`ps aux`、`netstat`、`crontab -l` 等信息。

#### Web 根目录加固

说明：

- 会把目录权限收敛到 `755`，文件权限收敛到 `644`。

#### 上传目录只读

说明：

- 会尝试处理 `upload`、`uploads`、`cache`、`tmp`、`temp` 等目录，把权限改成只读。

#### PHP 加固

说明：

- 会在站点目录写入 `.user.ini`，禁用 `system/exec/shell_exec/passthru/proc_open/popen/assert`。

#### 简易 WAF

说明：

- 会上传 `waf.php`，并通过 `.user.ini` 的 `auto_prepend_file` 挂载。
- 会拦截 `base64_decode`、`assert(`、`eval(`、`shell_exec`、`system(`、`passthru(` 等关键字。

### 6.8 数据库备份 / 恢复 / 改密

准备：

- SSH 链路可用。
- 数据库参数完整可用。
- 目标机上能执行 `mysql` 和 `mysqldump`。
- 目标机能访问 `database.host:database.port`。

说明：

- 数据库备份会把压缩后的 SQL 下载到 `output/backup/db/`。
- 数据库恢复会把备份重新上传到目标机后导入。
- 数据库改密成功后，程序会同步把 `config.yaml` 里的 `database.password` 更新成新密码。

### 6.9 批量 SSH 改密

准备：

- 目标列表。
- SSH 用户名。
- 一组旧密码候选。
- 新密码。
- 该账号有权执行 `chpasswd`，通常需要高权限账号。

说明：

- 如果输入框没填目标，会尝试读取 `output/target.txt`。
- 每台机器会依次尝试旧密码列表，成功后立即改成新密码。
- 成功主机会写到 `output/ssh_password_success.txt`。

### 6.10 PCAP 上传与检索

准备：

- `.pcap` 或 `.pcapng` 文件。
- 需要检索的关键字，例如账号、口令、路径、请求参数、域名等。

说明：

- 程序启动时会自动在本机监听 PCAP 上传服务，默认端口是 `18080`。
- 可上传到这些地址之一：
  - `http://127.0.0.1:18080/upload`
  - `http://127.0.0.1:18080/api/upload`
  - `http://127.0.0.1:18080/api/v1/upload`
- 表单文件字段名支持 `file`、`pcap`、`upload`。
- 可选字段 `client_id` 会进入索引，便于区分来源。
- 原始文件保存到 `pcap/`，索引写入 `bleve/`，明细写入 `pcap.sqlite`。

### 6.11 集成自检

准备：

- 当前机器有 Go 环境，`go build` 可用。
- 仓库里保留了 `cmd/integration-smoke` 源码。
- 如果你要完整跑通内置靶场回归，当前机器还要有对应的 WSL 靶场环境。

说明：

- 自检会先尝试重编译 `integration-smoke.exe`。
- 运行前会备份 `config.yaml` 和 `output/target.txt`，结束后自动恢复。
- 内置自检默认覆盖的链路包括：
  - 普通 Shell 执行
  - MD5 马上传与执行
  - Undead 马上传与执行
  - HTTP 取旗
  - Shell 取旗
  - Web 备份 / 查马 / 加固 / 恢复
  - SSH 改密
  - PCAP 上传与检索

## 7. 常见问题

### 界面能打开，但功能没结果

先查这几项：

- `output/target.txt` 有没有目标。
- `config.yaml` 里的 `shell.file`、`shell.pass`、`ssh.path`、数据库配置是否匹配真实环境。
- 目标是否真的在线，端口是否通。

### Shell 测试全失败

优先检查：

- Shell 地址是否拼对。
- Shell 参数名和请求方法是否匹配。
- 目标站点是否被 WAF、403、反向代理规则拦截。

### 防守类操作失败

优先检查：

- SSH 用户名/密码/端口是否正确。
- 当前账号是否有权限访问 `ssh.path`。
- 目标机是否存在 `tar`、`mysql`、`mysqldump`、`chpasswd`。

### PCAP 能上传，但搜不到

优先检查：

- 上传是否真的返回 `200 ok`。
- 稍等几秒再搜，索引是异步入库。
- 查询词是否真实存在于解析后的内容里。

## 8. 一个最小可用流程

如果你只是想先把流程跑通，按这个最小路径做：

1. 启动程序。
2. 配好 `config.yaml` 的 Shell 参数。
3. 在目标列表里填入目标 IP。
4. 先跑“存活探测”。
5. 再跑“Shell 测试”。
6. Shell 通了以后，再决定是否上传 Undead/MD5/Worm。
7. 需要值守时，再配置 SSH 和数据库功能。
8. 需要验证整套链路时，再跑“集成自检”。
